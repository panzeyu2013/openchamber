# PR: Unified Multi-Server Sidebar (#1412)

## Summary

Implements a unified multi-server sidebar that allows users to connect to multiple OpenCode server instances (local + remote SSH + remote URL) and manage sessions across them from a single sidebar. The architecture uses a server-side aggregation proxy (`MultiServerManager` + `SseFanIn`) that tags events with `serverId` and fans them into a unified event pipeline, eliminating the need for the UI to manage multiple SSE connections.

Single-server mode is the degenerate case — the UI path is identical whether there is 1 server or N servers, with no `isMultiServer` code fork.

---

## Design Overview

### Problem

OpenChamber currently supports only one OpenCode server instance. Users with multiple workstations or remote instances must open separate browser windows or VS Code instances. The sidebar, session list, chat, and event pipeline all assume a single server.

### Key Design Decisions

1. **ServerId is a first-class field** — every event carries `serverId` at top level, enforced by `ServerTagged<T>` type and `unwrapServerEvent()` which throws on missing `serverId`. No `|| 'local'` silent fallback in the unwrap path.

2. **Single server = degenerate case** — `SidebarServerHeader` gets `hidden` CSS when `servers.length <= 1`. `useServerSidebarSections` synthesizes a single `{ serverId: 'local' }` section when servers list is empty. No code fork.

3. **Nested Map stores** — `Map<serverId, Map<directory, Store>>` in both `ChildStoreManager` and `useGlobalSessionsStore`. Composite key `${serverId}::${directory}` in child stores.

4. **Two pipeline entry points with serverId guards** — `global-hub.js` `onEvent` (upstream SSE) and `feedEvent` (fan-in output) both guard against missing `serverId` by defaulting to `'local'`.

5. **Coalesce strategy** — Only `session.status` (per-session) and `message.part.delta` (per-message-per-part-per-field) are coalesced server-side within 16ms batches. Non-status events (`permission.asked`, etc.) dispatch immediately to prevent data loss.

6. **Circuit Breaker + Heartbeat** — `MultiServerManager` uses a circuit breaker (3 consecutive failures → open, exponential backoff capped at 60s) for REST-based `getGlobalSessions()`. `SseFanIn` has a dual-loop heartbeat (15s interval) that detects:
   - Subscribed `connecting` servers stuck without events → error after `connectTimeoutMs` (15s)
   - Non-subscribed `connecting`/`connected` servers without active subscription → disconnected after `heartbeatIntervalMs` (120s)
   - Subscribed `connected` servers with no events → disconnected after `heartbeatIntervalMs`

7. **Event recovery** — The SSE callback transitions ANY non-`connected` status back to `connected` when events resume, providing self-healing from heartbeat-triggered disconnect.

8. **Resource lifecycle** — `SseFanIn.destroy()` releases all resources (heartbeat timer, SSE subscriptions, flush timer, all internal Maps) and is wired into `gracefulShutdown`.

## Scope of Impact

### Directly impacted

- **Session sidebar**: Rendering now iterates `serverSectionsForRender` instead of flat `SidebarProjectsList`. Each server gets its own `DndContext`.
- **Chat header**: Shows `[server-label]` when multiple servers exist. Always visible, collapsible on click.
- **Chat input**: Shows server badge (icon + label) when `servers.length > 1`.
- **Global session list**: Fetches from `/api/servers/all/sessions` instead of a single-server endpoint. Returns nested `Map<serverId, Map<directory, Session[]>>`.
- **Event pipeline**: Every event now carries `serverId`. `unwrapServerEvent` throws on missing `serverId`.
- **Desktop host switcher**: Uses in-app `registerServer()` IPC instead of `window.location.assign()`.
- **Shutdown**: `SseFanIn.destroy()` releases all resources during graceful shutdown.

### Unchanged (verified)

- Single-server mode: `SidebarServerHeader` CSS `hidden`, chat header label returns `null`, chat badge gated on `servers.length > 1`.
- Session CRUD (create, rename, archive, delete): Unchanged paths.
- DnD, multi-select, folders: Unchanged.
- VS Code runtime: No changes (single-server by design, `ServerSection` fallback handles `servers.length === 0`).
- Mobile layout: Server header hidden in single-server mode, unchanged otherwise.

## Expected Behavior

### Single-server mode (local only)

- Sidebar, chat header, and chat input look and behave identical to before this PR.
- No visual changes, no new DOM elements visible.
- All existing features (search, rename, archive, delete, DnD, folders) work identically.

### Multi-server mode (local + remote)

- Sidebar shows a `SidebarServerHeader` for each server with:
  - Status dot: green (connected), yellow pulsing (connecting), gray (disconnected), red (error)
  - Hover-reveal: disconnect button (connected), reconnect button (disconnected/error)
  - Error/disconnected servers: reconnect button persistently visible, grayed-out sessions
  - Error state has tooltip with error message
  - Active server uses `text-foreground`, inactive uses `text-muted-foreground`
- Chat header shows `[server-label]` — always visible, click to collapse
- Chat input shows a server badge when multiple servers are connected
- Search shows "Searching N servers" hint
- Cross-server session switching: `useSessionActions.handleSessionSelect(sessionId, serverId)` routes to correct directory and server
- Server collapse state persisted to `localStorage`

### Error recovery

- Remote server SSE drop → heartbeat detects after 120s → marks disconnected → reconnect button visible → events resume → auto-transitions back to connected
- Server stuck in connecting → heartbeat detects after 15s → marks error with timeout message
- SDK import failure → server registered as error → user can retry

## Side Effects & Known Limitations

### Architectural

1. **Remote server REST registration returns 501** — `POST /api/servers` for remote URL servers is not yet implemented (SDK client creation deferred). Only the Electron IPC path can register remote servers today. Web/PWA/VS Code cannot register remote servers through the REST API.

2. **No automatic SSE reconnect for remote servers** — The heartbeat detects disconnections but does not automatically re-subscribe. Recovery requires user action (click "Connect") or external API call. The SSE callback does auto-recover status from `disconnected` → `connected` when events resume through an existing subscription.

3. **`Local` server string is not a centralized constant** — The string `'local'` appears in multiple files without a shared constant. Non-blocking tech debt.

4. **Coalesce for local events is passive** — The watcher guard (`if 'serverId' in payload return`) prevents local events from entering `sseFanIn.feedLocal()`. Local events rely on client-side coalescing only. This is correct for single-server but means local events have no server-side batch protection.

### UX

5. **REST API add-server path is non-functional for Web/PWA/VS Code** — The POST route returns 501. Adding servers from the Web UI requires the Electron desktop path or future SDK integration.

6. **Chat header server label is text-only** — No server status indicator in the header label area. Status is only visible in the sidebar server header.

7. **Error messages traverse 7 layers** — From `sse-fan-in.js` → `event-reducer.ts` → `sync-context.tsx` → `ServerInfo` → `ServerSection` → `SessionSidebar.tsx` → `SidebarServerHeader.tsx` tooltip. Any break in this chain would silently lose error details. Tests do not cover the full chain end-to-end.

### Performance

8. **`useGlobalSessionsStore.buildSessionsByDirectory`** rebuilds the nested Map on every session mutation. O(N) with N = session count. For typical workloads (100s of sessions), this is negligible.
